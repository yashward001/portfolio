## Problem

Desktop automation splits into two bad options: record-and-replay tools that shatter on any UI change, and scripting APIs that require a developer for every new workflow. Neither handles natural language, neither recovers gracefully from failure, and neither gives you any auditability over what ran and why.

## Constraints

- Intent arrives as voice or text and must map to typed, pre-approved UI actions — no ad-hoc scripting
- The model is only allowed to *select* capabilities, not generate arbitrary execution logic
- Every step must be observable: screenshot evidence, retry state, live status
- High-risk actions pause for explicit human approval before running
- Capability definitions must be storable and versionable, decoupled from prompts entirely

## Approach

Built across three deliberate layers. The frontend runs a formal 11-state/9-event state machine — `idle → listening → transcribing → planning → awaiting_approval → executing` — so every interaction phase is an explicit node, not a conditional buried in a component. Framer Motion drives the transitions.

The backend is structured around a Firestore capability registry: a schema-enforced document for every automatable UI action, each carrying its selector, typed parameters, preconditions, postconditions, risk level, confidence score, and auto-heal history. When a user speaks, Gemini 2.5 Flash receives the full registry as LLM tools and returns a plan as an ordered list of capability IDs with filled parameters. The model cannot generate Playwright code — it can only choose from pre-approved actions. This bounds the execution surface completely.

Playwright resolves each capability ID, executes with retry logic, captures a bounding-box screenshot, and pauses for approval if risk level demands it. A WebSocket bridge streams every status update to the frontend live.

Voice input bypasses a separate STT step entirely — Gemini Live API streams microphone audio directly, cutting latency between speaking and plan generation.

## Architecture

### Layer 1 — Frontend (Next.js + TypeScript)

A formal **11-state / 9-event state machine** governs every interaction phase:

- States: `idle`, `listening`, `transcribing`, `planning`, `awaiting_approval`, `executing`, `step_success`, `step_failed`, `retrying`, `completed`, `error`
- Events: `START_VOICE`, `TRANSCRIPT_READY`, `PLAN_READY`, `USER_APPROVED`, `STEP_DONE`, `STEP_FAILED`, `RETRY`, `ABORT`, `RESET`
- Framer Motion drives state-transition animations for immediate visual feedback

### Layer 2 — Backend (FastAPI + Firestore)

A **Firestore capability registry** stores every automatable UI action as a structured document with:

- `action_type`, `selector`, `description`, `parameters`
- `preconditions`, `postconditions`
- `risk_level` (low / medium / high)
- `confidence_score`, `auto_heal_history`

User intent is sent to **Gemini 2.5 Flash** alongside the full capability registry formatted as LLM tools. Gemini returns a structured execution plan as a sequence of capability IDs with filled parameters — no free-form scripting.

### Layer 3 — Execution (Playwright)

The Playwright engine runs each plan step sequentially:

- Resolves capability IDs against the Firestore registry
- Executes typed actions (click, fill, navigate, assert) with retry logic
- Captures per-step screenshots with bounding-box highlights
- Flags high-risk steps and pauses for human approval before proceeding

### Observability

A **WebSocket bridge** streams live execution state to the frontend:

- Per-step status updates (pending / running / success / failed / retrying)
- Screenshot evidence with annotated bounding boxes
- Retry indicators and approval gate prompts
- Full execution transcript on completion

### Voice-First Interaction

**Gemini Live API** streams microphone audio directly for real-time transcription and intent extraction, bypassing a separate STT step and reducing end-to-end latency from voice to plan generation.

## Key Technical Choices

1. **Capability registry over prompt-embedded selectors** — UI knowledge lives in Firestore, not in the prompt. Update a selector or add a capability without touching the inference layer or redeploying.
2. **Structured plan as capability IDs, not generated code** — the model's output surface is a typed list, not an arbitrary script. Every executable action was explicitly declared in advance.
3. **Formal state machine over ad-hoc flags** — 11 states, 9 events, zero undefined transitions. The state machine is independently testable and the transition graph is the documentation.
4. **Risk level in the registry, not inferred at runtime** — human approval gates are declared when a capability is authored, not decided by the model during execution.

## Results

- Full voice-to-execution pipeline operational across 6 applications (~50 capabilities): Gmail, Slack, Spotify, Notion, Outlook, Google Docs
- State machine eliminates undefined interaction states across the full session lifecycle
- Per-capability risk governance with confidence scores and auto-heal history in the registry
- Real-time WebSocket feed with annotated screenshots makes every step auditable after the fact

## What I'd Do Next

Automated capability discovery — crawl a target application's UI and propose new capability records rather than authoring them by hand. Regression harnesses that replay recorded execution traces against UI updates to detect selector drift. Policy-based approval routing so team deployments can define who approves what based on capability risk level, not just a binary gate.
